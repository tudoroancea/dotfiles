## Status model established by the generated integration

1. `agent_start` sets the parent agent to `working`.
2. `agent_end` schedules debounced `idle`; retryable provider errors remain `working` during the retry grace period and become `blocked` only if the retry does not resume.
3. `herdr:blocked` with `{ active: true, label }` increments a nesting counter and overrides both `working` and `idle` with `blocked`.
4. `herdr:blocked` with `{ active: false }` decrements that counter and restores the underlying state (`working`, retry hold/failure, or `idle`).
5. The integration reports `idle`, not `done`. Herdr derives the visible `done` state from an unseen idle completion, so our extensions must not try to publish `done` themselves.
6. On session shutdown the integration releases its lifecycle authority.

This design makes balanced, nest-safe `herdr:blocked` events the only extension-side status override currently available. There is no equivalent extension event for forcing `working` while Pi is otherwise idle.

## Audit findings

### Correct as implemented

- **Agentflow foreground runs and waits:** foreground semantic/raw agent calls and `agentflow_wait` remain inside the parent tool execution, so the normal Pi agent lifecycle keeps Herdr `working`. No override is needed.
- **Agentflow background runs:** launching in background returns promptly. Once the parent settles, Herdr may become idle/done even while child runs continue; this is correct because the parent pane is available and is not waiting for the user. A completed child result is delivered only while the parent is idle as a triggering `followUp`, which starts a new parent agent lifecycle and returns Herdr to `working`.
- **Background process launches:** the same rule applies. Active commands are intentionally represented by the extension footer status, not by holding the parent Herdr state at `working`.
- **Background completion/event delivery:** idle deliveries use triggering `followUp`; active deliveries use `steer`. The next/continuing parent turn owns `working`. Idle checks, delivery claims, and `agent_settled` flushing prevent duplicate notifications and false terminal transitions.
- **Explicit waits:** `background_wait` remains a running tool call, so Herdr stays `working`; timeout/cancellation returns through normal agent lifecycle handling.
- **Retry and shutdown paths:** the generated integration already handles retry grace, blocked precedence, idle debounce, and session release correctly.
- **Immediate custom UI setup:** `claude-style.ts` calls `done()` synchronously and does not await human input, so it must not emit `blocked`.
- **Automatic session naming:** naming runs after `agent_settled`, but it is an auxiliary side task and does not prevent use of the parent pane. Leaving Herdr idle/done is appropriate.
- **Notification timers and other passive UI/status extensions:** they do not require user input and should not claim `blocked`.

### Incorrect or incomplete

1. **Questionnaire modal is reported as `working`, not `blocked`.**
   - The tool awaits `ctx.ui.custom()` while the parent turn remains active.
   - No extension currently emits `herdr:blocked`.
   - Expected transition: `working → blocked → working → idle/done`, including answer, cancellation, thrown UI error, and abort paths.
   - The tool also checks only `ctx.hasUI`. Pi RPC mode has `hasUI === true`, but `custom()` is TUI-only and returns `undefined` in RPC, so the current code can dereference an undefined result instead of returning a clear unsupported-mode result.

2. **Agentflow dashboard is reported as `idle` while waiting for dashboard input.**
   - `/agentflow` is an idle command that awaits a custom dashboard.
   - The whole dashboard lifetime needs one blocked scope. Its nested cancellation confirmation must remain inside that existing scope rather than incrementing a second scope unnecessarily.

3. **Background task dashboard is reported as `idle` while waiting for dashboard input.**
   - `/background-tasks` has the same issue and the same nested-confirm behavior.

4. **Tool settings dialog is reported as `idle` while waiting for input.**
   - `/tools` awaits a `SettingsList` custom UI and needs a blocked scope.
   - It should reject or clearly handle non-TUI invocation before opening the blocked scope because `custom()` is not interactive in RPC mode.

5. **Session breakdown result viewer is reported as `idle` while waiting for input.**
   - The first `BorderedLoader` performs autonomous computation and must **not** be labeled blocked merely because Escape can cancel it.
   - The second custom UI is a result viewer that waits for user navigation/closure and does need a blocked scope.
   - RPC currently follows the `hasUI` branch even though `custom()` is TUI-only; non-TUI modes should use the existing non-interactive summary path.

### Known API limitation, not an extension bug to mislabel

An idle extension command performing autonomous work (for example the session-breakdown loader) cannot currently ask the generated integration to publish `working`; its private event contract only exposes `herdr:blocked`. Marking such work `blocked` would be semantically wrong. The plan therefore leaves autonomous side work idle unless Pi/Herdr later provides a general extension-state API. The future-maintenance rule in `AGENTS.md` requires this limitation to be reconsidered whenever these workflows change.

## Acceptance criteria

- Every extension-owned UI that genuinely requires human input publishes a balanced blocked scope.
- Cancellation, errors, and aborts cannot leave Herdr stuck in `blocked`.
- Foreground tools remain `working`; background work does not falsely hold the parent pane in `working` or `blocked`.
- Follow-up deliveries still create normal working/idle lifecycle transitions and remain deduplicated.
- Non-TUI modes never enter a TUI-only blocked wait or dereference `custom()`'s RPC default result.
- No extension reports `done` directly; Herdr continues deriving it from an unseen idle completion.
- The autogenerated Herdr extension is not modified.
